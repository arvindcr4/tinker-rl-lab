% Abstract

\begin{abstract}
  We investigate how Group Relative Policy Optimization (GRPO) behaves for small language models (0.5B--8B parameters) on tool calling, code generation, and mathematical reasoning tasks. Across 17 cloud GPU training runs on Tinker and multiple team-executed experiments, we observe strong gains on task-specific metrics: JSON tool-call validity improves from 0\% to 92\% and multi-turn tool chaining quality from 0.72 to 0.91 under a custom internal evaluation protocol, while code generation on a \emph{50-problem subset} of HumanEval improves from 32\% to 40\%. On GSM8K math reasoning, we conduct multi-seed replication of \emph{training-set reward} (mean last-10 reward 30.5\%, SD = 3.3\%, 5 seeds) and a LoRA rank ablation (rank 8--64). We identify seven original findings including a capacity threshold between 3B and 4B parameters, MoE routing-induced training volatility, a two-phase learning progression, and a 3--8$\times$ difficulty gap between synthetic and real-world tool schemas. Because held-out GSM8K evaluation is still pending, tool-calling uses custom rather than standardized evaluation, and code generation covers only a 50-problem subset, we frame these as training-dynamics evidence rather than generalization claims. These results position GRPO as a practically lightweight setup for studying post-training dynamics on verifiable tasks rather than claiming standardized cross-benchmark generalization.
\end{abstract}

\section{Introduction}

Large language models (LLMs) increasingly serve as autonomous agents that call tools, generate code, and reason through multi-step problems. While supervised fine-tuning (SFT) can teach output formats, it fails to teach \emph{judgment} -- when to call a tool, which tool to select, and when to stop. Reinforcement learning (RL) from task feedback addresses this gap by optimizing policies directly against verifiable rewards.

Group Relative Policy Optimization (GRPO) \citep{shao2024deepseekmath} is a critic-free variant of Proximal Policy Optimization (PPO) that computes advantages by normalizing rewards within groups of sampled completions. It requires no value function, no reference model for KL regularization, and substantially less compute than standard PPO -- making it attractive for resource-constrained post-training of small models.

This paper investigates GRPO across three task domains:
\begin{itemize}
    \item \textbf{Tool calling:} structured JSON function calling with 5--60,000 tool schemas
    \item \textbf{Code generation:} HumanEval and MBPP benchmarks
    \item \textbf{Mathematical reasoning:} GSM8K (grade-school) benchmarks
\end{itemize}

We execute experiments across model sizes from 0.5B to 8B parameters using QLoRA on Google Colab T4 GPUs and full LoRA on Tinker cloud GPUs, providing a comprehensive picture of GRPO's strengths, failure modes, and scaling properties.

Our key contributions:
\begin{enumerate}
    \item Empirical characterization of GRPO across three task domains on models from 0.5B to 8B parameters
    \item Multi-seed replication on GSM8K (5 seeds, mean 30.5\%, SD = 3.3\%) providing replication evidence
    \item LoRA rank ablation (rank 8/16/64) mapping the parameter-efficiency frontier for GRPO
    \item Seven original findings on capacity thresholds, MoE volatility, learning phases, and reward design
    \item Open-source Tinker SDK pipeline enabling GRPO on 8B models without local GPU resources
\end{enumerate}

\section{Background}

\subsection{Group Relative Policy Optimization (GRPO)}

GRPO \citep{shao2024deepseekmath} simplifies PPO by eliminating the critic network. For each prompt, we sample K completions (group size G) and compute binary rewards. Advantages are normalized within each group:

\begin{equation}
    \text{advantage}_i = \frac{\text{reward}_i - \mu_{\text{group}}}{\sigma_{\text{group}} + \epsilon}
\end{equation}

The policy gradient loss is:
\begin{equation}
    \mathcal{L} = -\mathbb{E}\left[\text{advantage}_i \cdot \sum_t \log \pi_\theta(a_t | s_t)\right]
\end{equation}

When all completions receive identical rewards (all correct or all incorrect), advantages are zero and no gradient update occurs -- the "zero-loss" phenomenon we track in Section \ref{sec:capacity}.

DeepSeekMath reports GRPO improving an instruction-tuned 7B model from 82.9\% to 88.2\% on GSM8K and 46.8\% to 51.7\% on MATH with group size G=64.

\subsection{Supervision Baselines}

Step-DPO \citep{lai2024stepdpo} demonstrates $\sim$+3\% MATH gains for >70B models with as few as 10K step-wise preference pairs. Li et al. show LLaMA-2-7B reaching 82.6\% GSM8K with synthetic SFT at $\sim$10$^6$ examples. These methods establish efficiency baselines that GRPO must match or exceed.

\section{Experimental Setup}

\subsection{Training Infrastructure}

\textbf{Tinker SDK (Cloud GPU):} We use Tinker v0.16.1 with custom GRPO loss via \texttt{forward\_backward\_custom}. Optimizer: Adam ($\beta_1$=0.9, $\beta_2$=0.95, $\epsilon$=1e-8). Runs write Tinker-hosted checkpoints, though recovery depends on run retention and account access at evaluation time.

\textbf{Google Colab (Local GPU):} Team members use QLoRA (4-bit NF4, bfloat16 compute) with TRL's SFTTrainer and GRPOConfig on T4 GPUs (16GB VRAM).

\textbf{Hyperparameters (Tinker runs):}
\begin{itemize}
    \item Batch size: 2 prompts per step
    \item Group size: G $\in$ \{4, 8, 16\}
    \item Max sequence: 1024 tokens (input), 512 tokens (generation)
    \item LoRA rank: 8, 16, or 32
\end{itemize}

These choices prioritize low-cost iteration rather than maximal stability. In particular, we do \emph{not} use KL regularization to an SFT reference policy and we do \emph{not} log policy entropy trajectories. We therefore interpret the reported runs as an initial empirical study rather than a fully drift-controlled RL recipe; lightweight KL anchoring and entropy diagnostics are important next-step mitigations.

\subsection{Decoding and Evaluation Protocol}

All GRPO training uses temperature $T=0.8$--$1.0$ with top-$p$ sampling for rollout generation. Reported ``accuracy'' values are binary reward means (1.0 = correct, 0.0 = incorrect) computed on fresh rollouts at the training temperature, \emph{not} greedy-decoded held-out test accuracy. For future held-out evaluation, we specify: greedy decoding ($T=0$), max 512 tokens, Qwen chat template, with pass@1 and bootstrapped 95\% CIs over the full test set.

\subsection{Reward Functions}

\textbf{Tool calling (3-component, 0--1):}
\begin{itemize}
    \item +0.3: Valid JSON output
    \item +0.4: Correct tool name
    \item +0.3: All argument keys present
\end{itemize}

\textbf{Math reasoning (binary):}
\begin{itemize}
    \item 1.0: Correct answer in \texttt{\textbackslash boxed\{\}} or as final number
    \item 0.0: Incorrect
\end{itemize}

\textbf{Code generation:} Binary pass/fail on test cases.

The SFT baseline's 0\% JSON validity is explained by the training data composition: the 500 SFT examples from Glaive-v2 included tool-use demonstrations, but the 1.5B model lacked the capacity to generalize the JSON format from limited supervised examples with standard chat-template prompting (Qwen \texttt{<|im\_start|>} format) and greedy decoding. GRPO's reward signal (0.3 for valid JSON + 0.4 for correct tool + 0.3 for arguments) provided the direct optimization pressure that SFT's cross-entropy loss could not.

Using Qwen2.5-3B-Instruct with SFT followed by GRPO (40 steps, LoRA rank 8):

\begin{table}[htbp]
    \centering
    \caption{Multi-turn tool chaining results (Qwen2.5-3B)}
    \begin{tabular}{lccc}
        \toprule
        Scenario & SFT & GRPO & Winner \\
        \midrule
        Weather + Packing & 0.90 & 0.90 & Tie \\
        Stock + News Chain & 0.77 & 0.90 & GRPO \\
        Search + Calculate & 0.63 & 0.92 & GRPO \\

    \label{tab:multi_turn}
\end{table}

Key finding: The -0.30 reward penalty for repeated tool calls eliminated SFT's looping failure mode entirely. The penalty triggers when the model generates a tool-call JSON with the same function name as any previous turn in the same episode; it is applied additively to the turn's reward. This mechanism prevents the model from repeatedly calling the same tool (e.g., infinite \texttt{get\_weather} loops) without suppressing legitimate multi-step sequences that invoke \emph{different} tools in succession.

Important caveat: these 0.72--0.92 multi-turn scores are \emph{custom reward-derived scenario scores}, not results from a standardized benchmark or proxy-state evaluator. We used a small hand-authored scenario set with deterministic rubric components, and inter-rater reliability was not measured because the scoring was not performed by multiple human annotators. We therefore present these numbers as internal diagnostics rather than definitive cross-paper benchmarks.

\subsection{Code Generation}

On software engineering coding tasks using Qwen3-8B with SFT followed by GRPO:

\begin{itemize}
    \item HumanEval pass@1: 32\% $\rightarrow$ 40\% (+8\%)
    \item Problems solved: 16/50 $\rightarrow$ 20/50 (+4)
    \item Training loss: 0.85 $\rightarrow$ 0.18 (-78\%)
\end{itemize}

These code-generation numbers come from a \textbf{50-problem subset} rather than the full 164-problem HumanEval harness. The subset was chosen to enable rapid iteration during the team's initial exploration of GRPO for code tasks (reducing evaluation time from $\sim$40 minutes to $\sim$12 minutes per checkpoint on a T4 GPU). We do not claim statistical significance or direct comparability to papers reporting full-suite pass@$k$. A full standard harness evaluation (HumanEval-164/MBPP with bootstrapped 95\% CIs and controlled temperature/seed) remains future work (Section~\ref{sec:future_plan}, Phase 3).

\section{Analysis}

\subsection{Capacity Threshold for GRPO}
\label{sec:capacity}

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth]{fig1_capacity_threshold.pdf}
    \caption{Capacity threshold for GRPO on GSM8K. (a) Training-set reward trajectories: the 4B model (green) starts high and saturates, while 8B models (blue, 5 seeds) show noisy but productive learning. The 3B model (not shown; stays at $\sim$2\%) fails entirely. (b) Zero-loss vs.\ zero-reward step percentages reveal opposite failure modes: 3B's zero-loss stems from all-incorrect groups (stalled), while 4B's stems from all-correct groups (saturated).}
    \label{fig:capacity}
\end{figure}

A capacity effect exists between 3B and $\geq$4B parameters for GRPO on GSM8K. Dense 3B models (Llama-3.2-3B) fail to learn, achieving only 0.78\%$\rightarrow$2.34\% training-set reward over 50 steps, while 8B models improve substantially. Notably, a Qwen3.5-4B model succeeds strongly: it starts at 87.5\% reward (step 1) and reaches 100\% repeatedly, with last-10 average of 82.5\%---narrowing the threshold to between 3B and 4B rather than 3B and 8B. The 3B failure traces to an inability to generate differential rewards within GRPO groups: 56\% of training steps had zero loss because all completions were equally wrong (zero-reward groups). By contrast, the 4B model's 68\% zero-loss rate stems from the opposite regime---all-\emph{correct} groups (0\% zero-reward)---indicating productive saturation rather than stalled training.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth]{fig2_diagnostics.pdf}
    \caption{GRPO training diagnostics on arithmetic (8B model, 25 steps). (a) Policy entropy collapses by $>$3 orders of magnitude, indicating convergence to a near-deterministic policy. (b) Group composition shifts from mostly mixed (gradient signal) to all-good (saturated, diminishing returns). (c) Reward rapidly approaches 1.0. These diagnostics were logged only for this task; comparable metrics for GSM8K/MATH remain a priority gap.}
    \label{fig:diagnostics}
\end{figure}

We quantify the failure mechanism via group-composition analysis from available training logs (Figure~\ref{fig:diagnostics}). On an arithmetic task with rich telemetry, an 8B model shows rapid progression from 81\% mixed groups (step 0) to 96\%+ all-good groups (step 4), with frac\_all\_bad never exceeding 1\%. Policy entropy collapses from 0.773 to 0.0002 over 20 steps, indicating convergence. By contrast, 3B models on GSM8K show 56\% zero-reward steps (all completions wrong $\Rightarrow$ zero within-group variance $\Rightarrow$ skipped updates), with up to 5 consecutive zero-signal steps.

However, this should be read as a \emph{capacity-plus-exploration} observation rather than a pure parameter-count theorem. The 3B failure may be confounded by reward sparsity, small group size (G=4--16), and limited exploration under fixed temperature. The decisive rescue ablations---larger group size (e.g., G=32--64 as in DeepSeek-style setups), broader temperature sweeps ($T \in \{0.7, 1.0, 1.3, 1.5\}$), and curriculum learning (easy-to-hard schedules)---were not run, so the present evidence does not isolate capacity from exploration effects. Literature suggests curriculum strategies consistently help sub-7B models overcome cold-start barriers \citep{parashar2025curriculum, chen2025selfevolving}.

\subsection{Synthetic vs. Real Data Gap}

On tool calling, synthetic 5-tool tasks saturate to reward >0.9 within 5 GRPO steps. Salesforce xlam-60k with diverse real-world tool schemas yields rewards of 0.06--0.36 after 100 steps -- a 3--8x difficulty gap.

\subsection{Two-Phase Learning Progression}

On tool-calling tasks, GRPO training exhibits a characteristic two-phase pattern:
\begin{itemize}
    \item \textbf{Phase 1 (Steps 1--20):} Model learns answer FORMAT compliance (0\%$\rightarrow$14\% accuracy)
    \item \textbf{Phase 2 (Steps 21--25):} Once format stabilizes, reasoning capability rapidly improves (14\%$\rightarrow$58\%)
\end{itemize}

This is analogous to curriculum learning without explicit curricula. However, we note that the two-phase pattern is most pronounced in tool-calling (where format compliance is a distinct, measurable sub-task) and in the arithmetic task (where format validity and correctness are separately tracked). On GSM8K, the pattern is less clear: breakpoints (first step with $>$50\% reward) vary from step 1 to step 10 across 5 seeds, and learning trajectories are noisy enough that a clean phase separation is not consistently identifiable. We therefore describe this as a \emph{domain-dependent} observation rather than a universal GRPO property.

\subsection{Reward Function Limitations}

Our reward functions are deliberately simple and verifiable, but this coarseness introduces known limitations. The tool-calling reward checks JSON validity, tool name, and argument-key presence but does \emph{not} verify argument-value correctness---a model can receive full reward by calling the right function with plausible but wrong values. This gap creates a reward-hacking surface: models may learn to produce syntactically correct but semantically incorrect tool calls. Adding argument-type and value checks, or leveraging outcome reward models such as ToolRM \citep{wang2025toolrm}, would reduce this risk. The math reward is binary (correct/incorrect) with no partial credit for intermediate reasoning steps, which contributes to reward sparsity for weaker models.

\subsection{Safety Considerations}

GRPO-trained tool-calling models introduce safety risks that warrant explicit discussion. First, \textbf{incorrect tool execution}: a model that reliably produces valid JSON may still call the wrong function or supply dangerous arguments (e.g., deleting data instead of reading it). Our reward function does not penalize semantically harmful tool calls. Second, \textbf{persistent loops}: although the -0.30 penalty for repeated tool calls eliminated looping in our experiments, more complex multi-step workflows may require explicit loop-detection safeguards. Third, \textbf{hallucinated capabilities}: models may generate calls to tools that do not exist in the available schema, particularly when transferring from synthetic to real-world tool sets. Future work should incorporate hallucination and loop penalties validated by standardized evaluators, and evaluate safety under adversarial prompting.

Across all domains, our reporting should be interpreted with the underlying data splits and compute budgets in mind. Tool and GSM8K numbers are primarily computed on training-prompt distributions with fresh rollouts; only the preliminary code-generation result uses a held-out benchmark subset. Typical Tinker runs use batch size 2, group size 4--16, 30--100 optimization steps, and cloud A100 time on the order summarized in the appendix. We do not yet report total tokens processed per run or full GPU-hours per experiment, which is another limitation for strict efficiency comparison.

\section{Related Work}

\paragraph{GRPO and critic-free RL.}
GRPO was introduced by \citet{shao2024deepseekmath} who achieved 88.2\% on GSM8K with a 7B model using group size G=64. Several contemporary methods also eliminate the critic: RLOO \citep{ahmadian2024backtobasics} uses a leave-one-out baseline and matches PPO in stability for some RLHF tasks, while REINFORCE++ \citep{hu2025reinforceplusplus} adds KL-shaped, z-scored advantages. Step-wise variants address GRPO's coarse sequence-level credit assignment: S-GRPO \citep{shen2025sgrpo} down-weights imbalanced groups to reduce noise, and StepGRPO \citep{huang2025stepgrpo} introduces dense intermediate rewards per reasoning step. We do not yet compare empirically to these alternatives; doing so is a priority for future work.

\paragraph{Preference-based methods.}
DPO \citep{rafailov2024dpo} and Step-DPO \citep{lai2024stepdpo} provide stable offline optimization without online rollouts. Step-DPO demonstrated $\sim$+3\% MATH gains for $>$70B models with as few as 10K step-wise preference pairs. \citet{li2024common7b} showed LLaMA-2-7B reaches 82.6\% GSM8K with synthetic SFT alone. These methods establish efficiency baselines that calibrate whether GRPO's online exploration is necessary at small scale.

\paragraph{Tool-use evaluation.}
Tool-calling evaluation has advanced rapidly: ToolRM \citep{wang2025toolrm} provides outcome reward models for structured tool-call assessment, and FC-RewardBench \citep{chen2025fcrewardbench} offers pairwise discrimination benchmarks for single-turn function calling. For multi-turn evaluation, proxy-state frameworks \citep{liu2025proxystate} judge tool-chaining quality via final-state inspection with LLM judges, providing hallucination diagnostics absent from our custom rubric. Our tool-calling scores (0.72--0.92) use a hand-authored scenario set without these standardized evaluators; adopting them would strengthen reproducibility.

\paragraph{Curriculum and exploration for small models.}
Curriculum-learning strategies consistently help sub-7B models overcome cold-start and sparse-reward barriers: easy-to-hard schedules \citep{parashar2025curriculum}, adaptive bandit curricula \citep{chen2025selfevolving}, and progressive length constraints \citep{hammoud2025trainlong} all yield measurable gains over random training. These approaches are directly relevant to our 3B failure mode (Section~\ref{sec:capacity}).

\paragraph{Parameter-efficient fine-tuning.}
Our QLoRA baseline \citep{dettmers2023qlora} enables single-GPU training via NF4 quantization. Newer methods target different efficiency frontiers: QR-Adaptor \citep{zhou2025qradaptor} jointly optimizes per-layer bitwidth and LoRA rank (+4.89\% GSM8K over prior quantized-LoRA), while LoTA-QAF \citep{chen2025lotaqaf} provides losslessly mergeable ternary adapters with 1.7--2.0$\times$ inference speedups. We retain QLoRA as our primary low-friction choice but note that a bounded comparison to these alternatives would sharpen efficiency claims.

Our work extends these findings to smaller models (0.5B--8B) and multiple task domains, providing systematic empirical characterization of GRPO's failure modes and scaling properties. Our current positioning is partial: we study a practical GRPO pipeline but not yet against the full contemporary comparison set outlined above.

\section{Planned Experiments and Standardized Evaluation Roadmap}
\label{sec:future_plan}

We outline a concrete experimental plan to address the evaluation and ablation gaps identified above, organized by priority.

\paragraph{Phase 1: Held-out evaluation (immediate).}
(a)~Recover trained LoRA adapters from Tinker checkpoints and evaluate on the full GSM8K test set (1,319 examples) with greedy decoding, reporting pass@1 with bootstrapped 95\% CIs over 3+ seeds.
(b)~Evaluate on MATH test set (5,000 examples) for transfer assessment.
(c)~Relabel all existing figures as ``training-set reward'' and add paired test-set plots.

\paragraph{Phase 2: Exploration and sparsity ablations.}
(a)~Group size sweep: $G \in \{16, 32, 64, 128\}$ on GSM8K for both 3B and 8B models, measuring zero-advantage fraction and reward density per group size.
(b)~Temperature sweep: $T \in \{0.7, 1.0, 1.3, 1.5\}$ to probe whether higher sampling temperatures rescue 3B learning.
(c)~Lightweight KL anchoring: $\beta \in \{0.01, 0.05, 0.1\}$ with logged KL/entropy trajectories.
(d)~Curriculum: easy-to-hard schedules on GSM8K sorted by solution length.

\paragraph{Phase 3: Standardized benchmarks and baselines.}
(a)~Tool calling: adopt FC-RewardBench-style pairwise discrimination and proxy-state final-answer evaluation with LLM judges; report judge reliability.
(b)~Code generation: run canonical HumanEval (164 problems) and MBPP harnesses with pass@1/pass@$k$ and 95\% CIs.
(c)~Baselines: add SFT-only, DPO, and (if feasible) RLOO/REINFORCE++ under matched compute budgets.

\paragraph{Phase 4: MoE and PEFT diagnostics.}
(a)~MoE routing telemetry: router entropy, per-expert load coefficient of variation, router-shift ratio.
(b)~LoRA rank ablation: $r \in \{8, 16, 32, 64\}$ with memory/throughput/accuracy reported side-by-side.
(c)~Bounded comparison to QR-Adaptor or LoTA-QAF on one model/task pair.

Estimated scope: $\sim$20--30 runs over 4--6 weeks, \$500--\$1000 in compute. Phases 1--2 can begin immediately from existing checkpoints and logs.

\section{Conclusion}

We presented an empirical study of GRPO for fine-tuning small language models (0.5B--8B) on tool calling, code generation, and mathematical reasoning. Our key findings, with scope qualifiers:

\begin{enumerate}
    \item \textbf{Tool calling:} GRPO transforms non-tool-calling models into reliable structured callers (0\%$\rightarrow$92\% JSON validity), with a -0.30 penalty eliminating persistent loops. \emph{Scope: custom internal evaluation; standardized benchmarks pending.}
    \item \textbf{Capacity--exploration interaction:} A sharp performance break exists between 3B and $\geq$8B parameters under our setup. The 3B failure stems from reward sparsity (56\% zero-reward steps $\Rightarrow$ zero gradient signal) rather than training instability, but this may be confounded by small group size and fixed temperature. \emph{Scope: not yet disentangled from exploration effects.}
    \item \textbf{Parameter efficiency:} LoRA rank 32 ($\sim$0.1\% trainable params) with QLoRA suffices for near-maximal training-set reward on GSM8K, with rank primarily affecting learning speed rather than asymptotic performance.
    \item \textbf{Synthetic--real gap:} A 3--8$\times$ difficulty gap exists between synthetic 5-tool schemas and diverse real-world tool schemas (xlam-60k), informing dataset design for tool-calling RL.
    \item \textbf{Training dynamics:} GRPO exhibits a two-phase progression (format compliance $\rightarrow$ reasoning), rapid entropy collapse ($>$3 orders of magnitude), and MoE routing-induced volatility (2.43$\times$ vs.~dense). \emph{Scope: training-set metrics only; held-out generalization unknown.}
\end{enumerate}

\paragraph{Limitations.} The most significant gap is evaluation: math and tool-use results are training-set reward metrics, not held-out test accuracy; code generation uses a non-standard 50-problem subset; and tool-calling evaluation relies on a custom rubric without standardized judges. Training runs omit KL regularization and entropy diagnostics, and the MoE analysis lacks routing-level telemetry. The concrete experimental roadmap in Section~\ref{sec:future_plan} addresses each of these gaps. In its current form, this work is best read as a practitioner-oriented study of GRPO's training dynamics and failure modes at small scale, not as a generalization claim.
