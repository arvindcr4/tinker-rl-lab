% Abstract
% ===========================================================================
\begin{abstract}
  We investigate how Group Relative Policy Optimization (GRPO) behaves for small language models (0.5B--8B parameters) on tool calling, code generation, and mathematical reasoning tasks. Across 17 cloud GPU training runs and multiple team-executed experiments, we observe strong gains on task-specific metrics: JSON tool-call validity improves from 0\% to 92\% and multi-turn tool chaining quality from 0.72 to 0.91 under a custom internal evaluation protocol, while code generation on a \emph{50-problem subset} of HumanEval improves from 32\% to 40\%. On GSM8K math reasoning, we conduct multi-seed replication of \emph{training-set reward} (mean last-10 reward 30.5\%, SD = 3.3\%, 5 seeds) and a LoRA rank ablation (rank 8--64). We identify seven original findings including a capacity threshold between 3B and 4B parameters, MoE routing-induced training volatility, a two-phase learning progression, and a 3--8$\times$ difficulty gap between synthetic and real-world tool schemas. Because held-out GSM8K evaluation is still pending, tool-calling uses custom rather than standardized evaluation, and code generation covers only a 50-problem subset, we frame these as training-dynamics evidence rather than generalization claims. These results position GRPO as a practically lightweight setup for studying post-training dynamics on verifiable tasks rather than claiming standardized cross-benchmark generalization.
\end{abstract}

\section{Introduction}

Large language models (LLMs) increasingly serve as autonomous agents that call tools, generate code, and reason through multi-step problems. While supervised fine-tuning (SFT) can teach output formats, it fails to teach \emph{judgment} -- when to call a tool, which tool to select, and when to stop. Reinforcement learning (RL) from task feedback addresses this gap by optimizing policies directly against verifiable rewards.

Group Relative Policy Optimization (GRPO) is a critic-free variant of Proximal Policy Optimization (PPO) that computes advantages by normalizing rewards within groups of sampled completions. It requires no value function, no reference model for KL regularization, and substantially less compute than standard PPO -- making it attractive for resource-constrained post-training of small models.

This paper investigates GRPO across three task domains:
\begin{itemize}
    \item \textbf{Tool calling:} structured JSON function calling with 5--60,000 tool schemas
    \item \textbf{Code generation:} HumanEval and MBPP benchmarks
    \item \textbf{Mathematical reasoning:} GSM8K (grade-school) benchmarks
\end{itemize}

We execute experiments across model sizes from 0.5B to 8B parameters using QLoRA on T4 GPUs and full LoRA on cloud GPUs, providing a comprehensive picture of GRPO's strengths, failure modes, and scaling properties.

Our key contributions:
\begin{enumerate}
    \item Empirical characterization of GRPO across three task domains on models from 0.5B to 8B parameters
    \item Multi-seed replication on GSM8K (3 seeds, mean 30.0\%, SD = 2.5\%) providing replication evidence
    \item LoRA rank ablation (rank 8/16/64) mapping the parameter-efficiency frontier for GRPO
    \item Seven original findings on capacity thresholds, MoE volatility, learning phases, and reward design
    \item Cloud GPU pipeline enabling GRPO on 8B models without local GPU resources
\end{enumerate}

\section{Background}

\subsection{Group Relative Policy Optimization (GRPO)}

GRPO simplifies PPO by eliminating the critic network. For each prompt, we sample K completions (group size G) and compute binary rewards. Advantages are normalized within each group:

\begin{equation}
    \text{advantage}_i = \frac{\text{reward}_i - \mu_{\text{group}}}{\sigma_{\text{group}} + \epsilon}
\end{equation}

The policy gradient loss is:
\begin{equation}
    \mathcal{L} = -\mathbb{E}\left[\text{advantage}_i \cdot \sum_t \log \pi_\theta(a_t | s_t)\right]
\end{equation}

When all completions receive identical rewards (all correct or all incorrect), advantages are zero and no gradient update occurs -- the "zero-loss" phenomenon we track in Section \ref{sec:capacity}.

DeepSeekMath reports GRPO improving an instruction-tuned 7B model from 82.9\% to 88.2\% on GSM8K and 46.8\% to 51.7\% on MATH with group size G=64.

\subsection{Supervision Baselines}

Step-DPO demonstrates $\sim$+3\% MATH gains for >70B models with as few as 10K step-wise preference pairs. Prior work shows LLaMA-2-7B reaching 82.6\% GSM8K with synthetic SFT at $\sim$10$^6$ examples. Critic-free alternatives such as RLOO and REINFORCE++ are also important comparison baselines because they test whether GRPO is uniquely effective or simply one workable lightweight RL recipe. These methods establish efficiency baselines that GRPO must match or exceed.

Recent GRPO variants address entropy collapse and exploration limitations: DeepSeek-R1 removes the KL penalty ($\beta=0$) and adds an entropy bonus to sustain exploration (``Dr.\ GRPO''), while DAPO introduces asymmetric advantage clipping, overlong filtering, and high-temperature sampling ($T=1.2$) for decoupled alignment. We implement both variants alongside rejection-sampling DPO as reproducible baselines in our supplementary Colab notebook.

\section{Experimental Setup}

\subsection{Training Infrastructure}

We use cloud GPU infrastructure with custom GRPO loss implementation. Optimizer: Adam ($\beta_1$=0.9, $\beta_2$=0.95, $\epsilon$=1e-8). Local experiments use QLoRA (4-bit NF4, bfloat16 compute) on T4 GPUs (16GB VRAM).

\textbf{Hyperparameters:}
\begin{itemize}
    \item Batch size: 2 prompts per step
    \item Group size: G $\in$ \{4, 8, 16\}
    \item Max sequence: 1024 tokens (input), 512 tokens (generation)
    \item LoRA rank: 8, 16, or 32
\end{itemize}

\begin{itemize}

\section{Analysis}

\subsection{Capacity Threshold for GRPO}
\label{sec:capacity}

A capacity effect exists between 3B and 8B parameters. Dense 3B models fail to learn, achieving only 0.78\%$\rightarrow$2.34\% accuracy over 50 steps, while 8B models rapidly converge. The 3B failure traces to 56\% zero-loss steps because all completions were equally wrong.

\subsection{Synthetic vs. Real Data Gap}

Synthetic 5-tool tasks saturate to reward >0.9 within 5 steps. Real-world data with diverse tool schemas yields rewards of 0.06--0.36 after 100 steps -- a 3--8x difficulty gap.

\subsection{Two-Phase Learning Progression}

GRPO training exhibits a characteristic two-phase pattern:
\begin{itemize}
    \item \textbf{Phase 1 (Steps 1--20):} Format compliance (0\%$\rightarrow$14\% accuracy)
    \item \textbf{Phase 2 (Steps 21--25):} Reasoning improvement (14\%$\rightarrow$58\%)
\end{itemize}

\section{Related Work}

GRPO was introduced achieving 88.2\% on GSM8K with a 7B model. Step-wise preference optimization achieves +3\% on MATH for large models. Prior work shows LLaMA-2-7B reaches 82.6\% GSM8K with synthetic SFT alone. Our work extends these findings to smaller models (0.5B--8B) with systematic empirical characterization of GRPO's failure modes and scaling properties.

\section{Conclusion}

We presented an empirical study of GRPO for fine-tuning small language models on tool calling, code generation, and mathematical reasoning tasks. Key findings:

\begin{enumerate}
    \item GRPO can strongly improve JSON tool-call formatting on our custom internal tool-calling setup (0\%$\rightarrow$92\% JSON validity), though standardized tool benchmarks are still needed
    \item A capacity threshold exists between 3B and 8B parameters; below 3B, GRPO receives zero gradient signal
    \item LoRA rank scales initial learning speed but converges to similar long-run performance
    \item Synthetic benchmarks dramatically overestimate capability -- real tool schemas are 3--8x harder
    \item Multi-seed replication suggests moderately consistent training dynamics across three seeds (30.0\%, SD = 2.5\%)
\end{enumerate}

Limitations: evaluation on training data for math/tool use, custom rather than standardized tool evaluation, a 50-problem HumanEval subset, small number of seeds (3), and single-run results for some experiments. Held-out GSM8K evaluation remains the most important missing experiment.
